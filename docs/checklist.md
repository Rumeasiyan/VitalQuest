## 1) Know exactly how you get 90+ (what markers reward)

Your marks are basically:

### Task 1: Implementation + User Manual = 50%

Markers grade this mostly from your **user manual**, not by checking the site.
Inside Task 1, you must hit:

- **10 pages = 10 marks** (1 mark per page)
- **Advanced/unique functionalities described clearly = 15 marks**
- **Visual design quality = 10 marks**
- **Limitations explained + desired final behaviour = 15 marks**

### Task 2: Show strategic + systems decisions in prototype = 25%

- **Exactly 5 examples**, **5 marks each**, screenshot-supported, must highlight differentiating features.

### Task 3: Cloud computing evaluation = 25%

- Pick a **real provider + real service**, justify, categorize (SaaS/PaaS/IaaS etc), discuss cost + security implications, compare to traditional, use literature.
- Mark split: 12 (selection/benefit) + 8 (categorisation/justification) + 5 (quality references).

# 2) The CW2 “90+ Task List” (follow in order)

## Phase A — Lock your CW2 scope (2–3 hours)

✅ **A1. Confirm the CW1 company you must build**

- CW2 must be “the same company outlined in group coursework.”
- From the CW1 file you attached now, the business appears to be **VitalQuest** (digital wellness / gamified platform).
    - If your CW1 is actually the cyber scanner startup, re-upload that CW1 so your CW2 stays consistent.

✅ **A2. Choose your “unique solution direction” (your CW2 uniqueness)**
Even if the business is the same, your solution must be **unique** and not group-built.
Pick ONE lens like:

- “Mobile-first onboarding + gamified retention loop”
- “Wearable integration flow + verification + anti-cheat”
- “Corporate wellness dashboard + B2B admin controls”
- “Privacy/compliance-first UX (GDPR/consent/controls)”
  (For cyber scanner CW1, the equivalent would be: VS Code extension / GitHub App / CI plugin / API-first etc.)

✅ **A3. Build your 10-page list (avoid About/Contact)**
Your prototype must have **10+ pages**, focusing on “important and interesting” pages and advanced features, not generic pages.

**Deliverable for A-phase:** a 1-page “Site Map & Core User Journey” doc.

## Phase B — Design a 10+ page prototype that screams “advanced features” (1–2 days)

✅ **B1. Create a navigation structure**

- Top nav: 5–7 items max
- Dashboard nav: sidebar tabs (this helps you easily reach 10+ pages without feeling forced)

✅ **B2. Define 2 user roles to make the site feel real**
Example patterns:

- User/Member
- Admin/Coach/Organisation manager
  This helps you create advanced pages (permissions, admin dashboards, reports).

✅ **B3. Write your “feature demonstration script”**
Because you can simulate functions (they don’t have to work fully).
Write a simple script like:

1. Create account → 2) Connect wearable → 3) Start quest → 4) Earn rewards → 5) View analytics → 6) Upgrade plan

✅ **B4. Build page wireframes (fast)**
Don’t design perfectly yet. Just layout:

- Inputs, buttons, tables, cards, steps, states (empty/loading/error/success)

✅ **B5. Apply a consistent design system**
To score the **visual design** marks.

- Consistent typography scale
- Consistent button styles
- Consistent spacing system (8px / 12px / 16px rhythm)
- Clean layouts (not overcrowded)

**Deliverable for B-phase:** first complete clickable prototype with all pages reachable.

## Phase C — Build the prototype (Google Sites or any tool) (1–2 days)

You’re advised to use Google Sites, but any tool is fine; marks aren’t for coding.

✅ **C1. Implement all pages**
Minimum = 10 pages.
**Important:** make each page look “complete” (real content, sample data, realistic UI blocks).

✅ **C2. Add “simulated” advanced functions**
You MUST show advanced business functionality, even if it’s not working in code.
Examples of simulation:

- “Scan results page” uses sample results
- “Checkout/upgrade flow” uses fake confirmation screen
- “Notifications settings” shows toggles + example alerts
- “Leaderboard” shows sample ranks + filter controls

✅ **C3. Add state screens**
This is an easy way to look advanced:

- success state
- error state
- empty state
- loading state

**Deliverable for C-phase:** final prototype ready for screenshots.

## Phase D — Screenshot pack (this directly earns marks) (3–5 hours)

Your user manual must include **screenshots of every implemented page**.

✅ **D1. Capture screenshots in a consistent way**

- Same browser width
- Same zoom
- Clear labels
- Don’t crop too aggressively (markers need context)

✅ **D2. Create a naming scheme**
Example:

- `P01_Landing.png`
- `P02_Onboarding.png`
- …
- `P10_AdminReports.png`

✅ **D3. Add “callouts”**
Use arrows / circles / numbers (PowerPoint is easiest) to point at the features you’ll explain.

## Phase E — Write the User Manual (max 1000 words) (1 day)

User manual is critical because they often mark Task 1 from it.
Also must include limitations + desired final behaviour.

✅ **E1. Use this structure (high scoring)**

1. **Purpose & target users** (2–3 lines)
2. **How to navigate** (menu/roles)
3. **End-to-end walkthrough** (your “demo script”)
4. **Page-by-page guide** (each page: what it is + what user can do)
5. **Advanced features explained** (the differentiators)
6. **Prototype limitations + final desired behaviour** (very explicit)

✅ **E2. For EACH page, include**

- Screenshot
- What the user sees
- What user does
- What happens next
- What is simulated (limitation) + how final system should behave

This directly targets:

- page marks + explanations
- limitations marks
- advanced functionality marks

## Phase F — Task 2: Five strategic/systems examples (max 700 words) (4–6 hours)

You must provide **five examples** connecting CW1 analysis/design → prototype pages, with screenshots.

✅ **F1. Select 5 “strong” CW1 items**
Pick items that produce visible features (markers prefer differentiating functionality).

Good sources to pick from CW1 usually include:

- SWOT strength/weakness mapped to a feature
- Porter force mapped to pricing/lock-in features
- PESTEL legal → consent/data controls
- Blue Ocean → unique experience/value innovation
- Use cases → exact screens you built

✅ **F2. Use this repeatable template (5 times)**
**Example #n (Title)**

- **CW1 insight:** (quote/summary + cite your CW1 section)
- **Design decision:** what you changed/implemented because of that insight
- **Prototype evidence:** screenshot + where it is in the site
- **Why it differentiates:** how it beats competitors
- **If changed from CW1:** what changed + why

**Deliverable for F-phase:** 5 mini writeups + 5 screenshots embedded.

## Phase G — Task 3: Cloud computing evaluation (max 700 words) (1 day)

You must choose a **specific cloud solution** from a **specific provider**, categorize it, discuss cost + security implications, compare vs traditional, and cite literature.

✅ **G1. Pick ONE “concrete” cloud solution**
Examples (choose one set that fits your business):

- AWS Amplify + Cognito + S3 + CloudFront (SaaS/PaaS mix)
- Azure App Service + Entra ID + Blob Storage
- Google Firebase (Auth + Firestore + Hosting) — often easy to justify for startups

✅ **G2. Categorise properly**
Explicitly state:

- Is it SaaS/PaaS/IaaS?
- Public/private/hybrid?
  …and justify.

✅ **G3. Evaluate integration in your business**
Write:

- What exact part of your system uses it (auth, storage, analytics, notifications, hosting)
- Data flow (what goes where)
- Operational benefit (scaling, reliability, speed)

✅ **G4. Cover cost + security implications**

- Cost drivers (traffic, storage, compute, auth users)
- Security controls (IAM, encryption, backups, logging, WAF, etc.)
- Risks (vendor lock-in, misconfigurations)

✅ **G5. Compare to traditional**

- On-prem server: CAPEX, maintenance, uptime burden
- Cloud: pay-as-you-go + elasticity

✅ **G6. Literature + references**
This is literally 5 marks.
Use: provider docs + 2–4 academic/industry sources.
